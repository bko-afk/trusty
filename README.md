# Trusty Insurance Reviews

Каталог туристических и медицинских страховых компаний с рейтингами, отзывами, жалобами, статьями, пользовательским кабинетом и Payload CMS.

Стек: Next.js App Router, React, TypeScript, Payload CMS, PostgreSQL и Tailwind CSS. Основной язык публичного сайта — английский (`/companies`); русская и испанская версии имеют отдельные индексируемые URL (`/ru/companies`, `/es/companies`).

## Локальный запуск

Требования: Node.js 20.9+ и доступная PostgreSQL.

```bash
cp .env.example .env
npm ci
npm run migrate
npm run dev
```

- Сайт: `http://localhost:3000`
- Админка: `http://localhost:3000/admin`
- Проверка типов: `npm run lint`
- Production build: `npm run build`

`npm run seed` предназначен только для первоначального заполнения пустой базы. Не запускайте его повторно на рабочей базе без резервной копии.

## Переменные окружения

Обязательные:

- `DATABASE_URI` — строка подключения PostgreSQL.
- `PAYLOAD_SECRET` — случайная строка длиной не менее 32 символов.
- `NEXT_PUBLIC_SERVER_URL` — основной публичный URL без завершающего `/`.

Опциональные:

- `S3_*` — S3-совместимое хранилище для загрузок из админки. На Vercel локальные файлы не сохраняются между деплоями, поэтому для новых загружаемых изображений нужен R2/S3.
- `RESEND_API_KEY` и `EMAIL_FROM_*` — доставка системных писем. Без них сайт работает, но отправка писем отключена.
- `ADDITIONAL_TRUSTED_ORIGINS` — дополнительные origin через запятую на время переноса домена.
- `UPSTASH_REDIS_REST_URL` и `UPSTASH_REDIS_REST_TOKEN` — общий rate limit для Vercel и нескольких экземпляров приложения. Без них используется только локальный резервный лимитер процесса.
- `RATE_LIMIT_SALT` — отдельная случайная строка для анонимизации сетевых идентификаторов; при отсутствии используется `PAYLOAD_SECRET`.

Никогда не добавляйте `.env` в Git или Docker-образ. `.dockerignore` уже исключает локальные секреты.

## Текущий деплой: Vercel

В Vercel должны быть заданы `DATABASE_URI`, `PAYLOAD_SECRET` и `NEXT_PUBLIC_SERVER_URL=https://trusty-blue.vercel.app`. Системные `VERCEL_PROJECT_PRODUCTION_URL` и `VERCEL_URL` используются автоматически для корректных canonical URL и preview-деплоев.

После изменения схемы Payload:

```bash
npm run migrate:create
```

Перед релизом сделайте резервную копию базы, проверьте статус и примените миграции до переключения приложения на новый код:

```bash
npm run migrate:status
npm run migrate
npm run migrate:status
```

В последнем статусе у каждой миграции должно быть `Ran: Yes`. Изменения схемы в этом проекте обратно совместимы с предыдущим релизом, поэтому безопасный порядок для Vercel: резервная копия, миграции, push/deploy, проверка `/`, `/companies`, `/admin`.

Обычные изменения React, CSS, переводов и SEO миграции БД не требуют.

## Админка и модерация

- `Companies` управляет публикацией, проверкой профиля, логотипом, позициями в общем рейтинге и позициями по видам страхования.
- `Reviews` и `Complaints` создаются со статусом `pending` и появляются публично только после подтверждения администратора.
- Каждый отзыв со статусом `published` автоматически влияет на рейтинг; остальные статусы исключаются из расчёта.
- `Site Settings` управляет блоками главной, подборками компаний и основными SEO-настройками.
- `Articles` управляет блогом, обложками и SEO статей.

У локализуемых полей компаний, видов страхования, статей и SEO в верхней панели редактора Payload доступен выбор `RU / EN / ES`. Существующий контент перенесён в RU; английские и испанские переводы следует заполнять в соответствующей локали, не изменяя общий `slug`.

Каждая публичная языковая версия получает собственные `lang`, canonical, Open Graph и взаимные `hreflang`; sitemap содержит все три URL и `x-default`. Адреса с лишним английским префиксом (`/en/...`) постоянно перенаправляются на основной URL без префикса.

Публичные формы используют отдельный `/api/submissions/*`, принимают только разрешённые поля, проверяют origin и размер запроса и имеют защиту от простого спама. Прямое публичное создание компаний, отзывов и жалоб через Payload REST закрыто.

## Переезд на Hostinger

Рекомендуемый вариант — Hostinger VPS с Docker Compose, Coolify или Dokploy. Код не зависит от Vercel API.

1. Создайте резервную копию Neon через `pg_dump`.
2. Разверните репозиторий на VPS и заполните `.env`.
3. Сначала можно оставить текущий Neon `DATABASE_URI`, перенеся только приложение.
4. Соберите образ и примените миграции:

```bash
docker compose build app
docker compose run --rm app npm run migrate
docker compose up -d
```

5. После проверки переключите DNS и замените `NEXT_PUBLIC_SERVER_URL` на настоящий домен.
6. Если переносите PostgreSQL на VPS, восстановите дамп в контейнер `db`, затем измените `DATABASE_URI` на адрес вида `postgresql://postgres:password@db:5432/insurance`.
7. Во время перехода добавьте старый и новый домены в `ADDITIONAL_TRUSTED_ORIGINS`, после завершения оставьте только основной.

Перед переключением домена проверьте `/`, `/companies`, формы, вход, `/admin`, загрузку изображения и отправку тестового отзыва на модерацию.

## Безопасность и обслуживание

- Production не стартует без `DATABASE_URI` и корректного `PAYLOAD_SECRET`.
- Админские и пользовательские входы ограничивают неудачные попытки.
- Публичные записи проходят модерацию и не могут самостоятельно менять рейтинг.
- API аккаунта использует same-origin проверки и ответы `private, no-store`.
- Заголовки CSP, HSTS, COOP, CORP, `nosniff`, frame protection и Permissions Policy задаются в Next.js.
- Регулярно выполняйте `npm audit --omit=dev`, обновляйте зависимости совместимыми версиями и делайте резервные копии PostgreSQL и S3.

Абсолютной защиты от взлома не существует. Для высокой нагрузки после переезда дополнительно включите WAF на уровне Cloudflare или хостинга. Встроенный лимитер становится распределённым при настройке Upstash и остаётся дополнительным слоем к WAF.
